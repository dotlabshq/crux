# Team Operations Principles

This reference supports `team-operations-coach`.

## Purpose

Use markdown as the primary coordination layer for teams.
Do not use it as a full replacement for a detailed ticket system.

## Core Rules

- Keep weekly notes short
- Make ownership explicit
- Use summary-first writing
- Surface blockers early
- Mark stale information clearly
- Track team focus, not every micro-task

## Operating Principles

### 1. Summary First

Every weekly note should start with a short summary before details.

### 2. Visible Ownership

Important work should have a visible owner.
If ownership is unclear, mark it as a risk.

### 3. Weekly Rhythm

The system works only if teams update it on a weekly cadence.
If a weekly note is missing or stale, report that explicitly.

### 4. Lean Visibility

Track:
- focus
- blockers
- dependencies
- carry-over
- risks

Avoid:
- long narrative status notes
- full ticket mirrors
- duplicated execution detail already stored elsewhere

### 5. Leadership Support

Use the operating notes to understand whether teams need:
- more direction
- more coaching
- more support
- more delegation

Explain this in plain language first.

### 6. Stale Data Handling

Mark a team view as stale when the weekly note is missing or outdated.
Do not silently present old status as current.
