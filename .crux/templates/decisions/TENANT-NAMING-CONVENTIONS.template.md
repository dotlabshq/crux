# Tenant Naming Conventions — Platform-Wide Standards

> Generated by Crux coordinator during installation.
> To change a pattern: propose an amendment, get user approval, update this file.
> All agents must lazy-load this file when handling tenant operations.

## Context

{{PROJECT_NAME}} is a multi-tenant platform. Tenants are isolated at:
{{ISOLATION_LAYERS}}
Without a shared naming standard, each layer develops its own conventions
independently, making tenant lookup, cross-layer debugging, and audit difficult.

## Decision

All tenant-scoped resources across all platform layers must follow the conventions
defined in this document. Each agent enforces its own layer but reads this
document as the normative source. Layer-specific operational docs are derived
from this decision and must not contradict it.

---

## Tenant Identity Rules

**Tenant ID**
- Format: `^[a-z0-9][a-z0-9-]{0,19}$`
- Lowercase alphanumeric, hyphens allowed, no leading hyphens
- Max 20 characters
- Must be unique across the platform

**Environment names**
- Canonical set: {{ENVIRONMENTS}}
- No abbreviations (not `prd`, `stg`, `development`)
- `{{PRODUCTION_ENV}}` is always production-sensitive regardless of agent

**Owner contact**
- Must be a valid email address
- Stored as `owner-email` label/annotation/comment on all tenant resources

### PostgreSQL — Schemas (schema-per-tenant)

Pattern: `{{POSTGRESQL_SCHEMA_PATTERN}}`

Note: PostgreSQL identifiers use underscores, not hyphens.
Hyphens in tenant-id are replaced with underscores for all PostgreSQL objects.

| Tenant | Env | Schema |
|---|---|---|
{{POSTGRESQL_SCHEMA_EXAMPLES}}

Required comment on schema:

### PostgreSQL — Tables

Tables live inside tenant schemas — no tenant prefix needed on table names.
Table names must be:
- `snake_case` only
- Descriptive, domain-relevant
- No abbreviations unless universally understood (e.g. `id`, `url`, `uuid`)
- Plural for entity tables: `users`, `orders`, `products`
- Singular for config/lookup tables: `config`, `setting`

Required columns on all entity tables:
```sql
id          uuid or bigserial   PRIMARY KEY
created_at  timestamptz         NOT NULL DEFAULT now()
updated_at  timestamptz         NOT NULL DEFAULT now()
```

- New agents have a clear, approved convention to implement
