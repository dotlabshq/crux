# codebase-scanner

**Owner**: `backend-developer`
**Type**: `read-only`
**Approval**: `No`

---

## What I Do

Scans the project directory structure, reads manifest files and key source
files, detects language, framework, conventions, and entry points, then
writes a structured architecture document to `.crux/docs/backend.md` with
a token-efficient summary at `.crux/summaries/backend.md`.

---

## When to Use Me

- `.crux/docs/backend.md` is missing and project directory is available
- User requests: "scan codebase", "generate architecture doc", "what's the stack"
- Stack or structure has significantly changed since last scan
- Onboarding Step 4 triggers me automatically

---

## Steps

```
Step 1 — Locate project root
  Look for manifest file in current and parent directories:
    package.json, go.mod, Cargo.toml, pyproject.toml,
    pom.xml, build.gradle, composer.json, mix.exs
  IF not found → stop, notify:
    "No project manifest found. Are you in the right directory?"

Step 2 — Read manifest
  Parse manifest file:
    name, version, language/runtime version
    main dependencies (top 10 by relevance)
    scripts (build, test, start, dev)

Step 3 — Map directory structure
  List directories at depth 1 and 2.
  Identify purpose of common dirs:
    src/ internal/ lib/ app/ cmd/ api/ pkg/     → source
    tests/ test/ spec/ __tests__/               → tests
    docs/ .crux/ README*                        → documentation
    db/ migrations/ prisma/ ent/                → database
    scripts/ Makefile                           → tooling
    config/ .env* docker*                       → configuration

Step 4 — Detect conventions (sample key files)
  Read up to 3 files from the main source directory:
    → identify naming patterns (camelCase, snake_case, kebab-case)
    → identify file organisation pattern (by layer, by feature, by type)
    → identify error handling pattern if visible
    → identify response envelope if API layer is present
  Keep samples short — extract patterns, do not store full file content.

Step 5 — Detect test structure
  Check test directory (if found):
    → test runner and config file
    → test naming convention
    → test/source ratio (rough estimate)

Step 6 — Detect API style (if applicable)
  Look for route definitions or controller files.
  Identify: REST / GraphQL / gRPC / other
  Identify: versioning pattern (/v1/, headers, etc.)
  Identify: auth pattern if visible (JWT, session, API key)

Step 7 — Write .crux/docs/backend.md
  Use the output format below.

Step 8 — Write .crux/summaries/backend.md
  Max 200 tokens. Use summary format below.

Step 9 — Update MANIFEST.md docs entry:
  last-updated: {DATE}
  tokens: {estimated}
  summary: .crux/summaries/backend.md

Step 10 — Skill complete
  Notify: "Architecture document written to .crux/docs/backend.md"
```

---

## Error Handling

| Condition | Action |
|---|---|
| No manifest file found | Stop. Ask user to confirm working directory. |
| Manifest found but unreadable | Note as `(unreadable)`, continue with directory scan. |
| Source files unreadable | Skip that directory, note in output. |
| `.crux/docs/` not writable | Write to `.crux/workspace/output/backend.md`. Notify user. |
